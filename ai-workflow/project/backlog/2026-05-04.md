## TASK-019 command/audit 최소 schema 및 risk mitigation command API 구현

- 상태: done
- 우선순위: high
- 담당: Codex
- 영향 문서/코드: `backend-core/migrations/`, `backend-core/internal/domain/`, `backend-core/internal/store/`, `backend-core/internal/httpapi/`, `docs/backend_api_contract.md`, `ai-workflow/memory/backend_development_roadmap.md`

### 1. 계획 (Plan)

- 작업 목표: risk mitigation을 즉시 boolean 결과가 아니라 command lifecycle과 audit log로 기록하는 1차 경로를 구현한다.
- 범위:
  - `commands`, `audit_logs` migration 추가.
  - risk mitigation command 생성과 idempotency 처리 구현.
  - command 상태 조회 API 추가.
  - API 계약과 workflow state 문서 갱신.

### 2. 실행 (Act)

- `000003_create_command_audit_tables` migration을 추가했다.
- `domain.Command`, `domain.AuditLog`, `domain.RiskMitigationCommandRequest`를 추가했다.
- `PostgresStore.CreateRiskMitigationCommand`, `PostgresStore.GetCommand`를 추가했다.
- `POST /api/v1/risks/{risk_id}/mitigations`를 추가해 `pending` command와 audit log를 생성하도록 했다.
- `GET /api/v1/commands/{command_id}`를 추가했다.
- `idempotency_key`가 기존 command와 일치하면 중복 생성 없이 기존 command/audit log를 반환하도록 했다.
- 홈랩 PostgreSQL `devhub` DB에 migration version `3`을 적용했다.

### 3. 검증 (Validate)

- `cd backend-core && go test ./...` 통과.
- `DEVHUB_TEST_DB_URL=... go test ./internal/store -run TestPostgresStoreProcessesNormalizedWebhookEvent -count=1 -v` 통과.
- `make migrate-version` 결과 `3`.

### 4. 결과 (Result)

- 작업 결과: risk mitigation 요청은 audit 가능한 command lifecycle로 들어가며, 프론트는 `command_id`를 기준으로 상태 조회를 시작할 수 있다.
- 다음 작업:
  1. service action command API 구현.
  2. command status transition worker 또는 executor boundary 설계.
  3. Gitea Runner 세부 상태 adapter 또는 Gitea REST client 연동 범위 확정.
  4. production command API에서는 `X-Devhub-Actor` header 대신 JWT/session 기반 actor verification을 적용.
