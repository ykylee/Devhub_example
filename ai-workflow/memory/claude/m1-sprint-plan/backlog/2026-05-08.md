### T-M1-01 — SEC-5 `writeServerError` 헬퍼 도입 (DoD #5)

- 우선순위: P1 (별도 cleanup PR)
- 위치:
  - 신규 헬퍼: `backend-core/internal/httpapi/errors.go` (또는 기존 `router.go` 의 helper section)
  - 영향 핸들러 (총 22 위치, 6 파일):
    - `audit.go:56`
    - `commands.go:118, 194, 232`
    - `domain.go:85, 122, 161, 203`
    - `events.go:58`
    - `organization.go:148, 200, 221, 263, 345, 776`
    - `snapshot.go:85, 111, 128, 145, 150, 177, 218, 237, 267, 290`
- 변경:
  - `func writeServerError(c *gin.Context, err error, op string)` — `log.Printf("server error: op=%s err=%v", op, err)` 후 `c.JSON(500, gin.H{"status":"failed","error":"internal error"})`.
  - 모든 `c.JSON(http.StatusInternalServerError, gin.H{"status":"failed","error": err.Error()})` 호출을 `writeServerError(c, err, "<op>")` 로 치환. `op` 는 핸들러명 + 상위 동작(예: `"organization.list_users"`).
  - response envelope 자체는 유지 (`{"status":"failed","error":"internal error"}`) — DoD #1 (envelope 정합) 와 충돌 없음.
- 의존: 없음 — 단독 cleanup PR 로 머지 가능.
- DoD:
  1. 22 위치 모두 헬퍼로 치환, 빌드/`go vet`/`go test ./...` PASS.
  2. 단위 테스트 1건: 헬퍼가 `err.Error()` 를 응답 본문에 노출하지 않고 `internal error` 로 마스킹하는지 확인.
  3. 회귀 테스트 — 기존 핸들러 테스트가 응답 본문의 `error` 필드를 *정확한 메시지로* 비교하던 케이스가 있다면 `internal error` 로 갱신.
- 위험:
  - 4xx 응답(`c.JSON(http.StatusBadRequest, ...)`) 은 의도적으로 사용자에게 의미 있는 메시지를 노출하므로 변경 대상이 아니다 — 헬퍼는 5xx 전용.
  - `errors.go` 의 helper 가 향후 `op` 외에 `request_id` 까지 받도록 확장 가능 → DoD #3 (audit actor 보강) 와 시너지.
- 규모: 작음 (~80 LoC + 테스트 1건)
- 대응 PR: **M1-PR-A** (SEC-5 cleanup, 단독)

## 4. 검증 절차

각 PR 머지 전 본 절차를 실행한다.

1. **Backend**: `cd backend-core && go test ./...`
2. **Frontend** (해당 PR 가 frontend 변경 포함 시): `cd frontend && npm run build`
3. **계약 정합** (PR-B): `docs/backend_api_contract.md` diff 와 핸들러/타입 정합 sub-agent 검증.
4. **마이그레이션** (PR-D): up → down → up 사이클 수동 확인.
5. **회귀**: 머지 후 `go test ./internal/httpapi/...` 풀 매트릭스 PASS 유지.
6. **마커 청소**: 본 sprint 종료 시 `// M1-TODO|// M1-LIFECYCLE|// M1-AUDIT` 임시 마커 0건.
