## 🔐 SEC-2 — Backend BearerTokenVerifier 미주입 + nil fallback

- 상태: planned
- 우선순위: high
- 책임자: backend-core
- 발견 경위: PR #12 종합 리뷰 (2026-05-08, claude code). Codex/self-review 에서는 다루지 않음.
- 위치:
  - `backend-core/internal/httpapi/auth.go` — `cfg.BearerTokenVerifier == nil` 분기에서 `X-Devhub-Auth: bearer_unverified` 헤더만 부착 후 통과.
  - `backend-core/main.go` — `RouterConfig` 생성 시 `BearerTokenVerifier` 필드를 전혀 채우지 않음. 즉 prod 빌드도 nil fallback 경로로 동작.
- 영향: Authorization: Bearer 헤더가 붙은 요청은 토큰 내용 무관하게 통과. organization / RBAC / commands 등 mutation API 가 사실상 무인증.
- 마감: SEC-1 과 동일 — Phase 5.2 종료 전.

### 완료 정의 (DoD)

1. `BearerTokenVerifier` 구현체 1종 이상 존재 (예: IdP introspection / JWT 검증). 위치 후보 `backend-core/internal/auth/`.
2. `main.go` 에서 `cfg` 또는 환경변수 기반으로 verifier 인스턴스 생성하여 `RouterConfig.BearerTokenVerifier` 에 주입.
3. `auth.go` 의 nil 분기를 제거하거나, dev 전용 환경변수(예: `DEVHUB_AUTH_DEV_FALLBACK=1`) 가 명시될 때만 허용. 기본 동작은 401.
4. config layer 에 fail-fast 가드 — prod 모드에서 verifier 가 nil 이면 startup 거부.
5. **`auth.go` 의 *empty Authorization → c.Next()* 분기 제거** (또는 prod 에서 401). 인증 불필요 라우트는 `/healthz`, `/api/v1/snapshot` 등 명시적 화이트리스트만 허용. *(2026-05-08 codebase 리뷰 보강)*
6. `auth_test.go` 에 prod 가드 테스트 케이스 추가 (env unset + verifier nil → fail-fast, 빈 Authorization → 401).

### 의존 / 결합

- SEC-1 의 frontend 토큰 발급 흐름과 컨트랙트 일치 필요. 동일 sprint 내 두 항목 함께 진행 권장.
- IdP (Hydra/Kratos) 연동 정책은 `ai-workflow/memory/claude/phase13/` 자료 참조.

## 🔐 SEC-3 — Backend role(권한) 미적용 (RBAC 매트릭스 unenforced)

- 상태: planned
- 우선순위: high
- 책임자: backend-core
- 발견 경위: 2026-05-08 코드베이스 전체 보안 리뷰(`codebase-security-review-2026-05-08.md` Vuln A 구성 요소 #3).
- 위치:
  - `backend-core/internal/httpapi/auth.go:73` — `devhub_actor_role` 을 gin context 에 set
  - `backend-core/internal/httpapi/router.go` — `/api/v1` 라우트 정의에 role 가드 미들웨어 부재
  - `backend-core/internal/httpapi/rbac.go` — `defaultRBACPolicy()` 매트릭스 정의 존재하지만 라우트 결정에 미연결
  - 모든 mutating 핸들러 (`organization.go`, `commands.go` 등) — `devhub_actor_role` context 값을 읽지 않음
- 영향: SEC-2 가 해소돼 verifier 가 주입되더라도, 발급된 토큰의 role 이 어떤 라우트에도 강제되지 않아 모든 인증 사용자가 동등 권한. RBAC 정책은 단지 *참조 자료* 로만 존재.
- 마감: SEC-2 와 동일 sprint. SEC-2 단독 해소는 권한 차등을 만들지 못함.

### 완료 정의 (DoD)

1. 라우트별 role 가드 미들웨어 1종 이상 (예: `requireRole(domain.AppRoleSystemAdmin)`) 구현 후 다음 라우트에 적용:
   - `/admin/*` → `system_admin`
   - 사용자 PATCH/DELETE (`PATCH/DELETE /api/v1/users/:user_id`) → `system_admin`
   - `GET /api/v1/audit-logs` → `manager` 이상
   - 조직 단위 CRUD (`POST/PATCH/DELETE /api/v1/organization/units...`, `PUT /api/v1/organization/units/:unit_id/members`) → `system_admin`
   - command 생성 (`POST /api/v1/admin/service-actions`, `POST /api/v1/risks/:risk_id/mitigations`) → `manager` 이상
2. role 가드 결과는 `defaultRBACPolicy()` 의 matrix 와 정합 (rank 비교) — 정적 enum 매핑 또는 정책 객체 주입 중 택1.
3. role 부족 시 403 응답 + `audit_logs` 에 거부 시도 기록.
4. 통합 테스트: 각 role × 각 보호 라우트 조합 → 허용/거부 매트릭스 검증.
5. `// SECURITY (SEC-3)` 주석 제거.

### 의존 / 결합

- SEC-2 의 verifier 가 actor role 을 정확히 채워야 가드가 의미 있음.
- RBAC write API (PR description 의 향후 작업) 와 정책 인스턴스 공유 검토.

## 🔐 SEC-4 — `X-Devhub-Actor` 헤더 기반 actor 스푸핑

- 상태: planned
- 우선순위: high
- 책임자: backend-core
- 발견 경위: 2026-05-08 코드베이스 전체 보안 리뷰(`codebase-security-review-2026-05-08.md` Vuln A 구성 요소 #4).
- 위치:
  - `backend-core/internal/httpapi/commands.go:271-287` — `requestActor` 함수의 `X-Devhub-Actor` 헤더 fallback
  - 호출처: `recordAudit` (audit.go), `createServiceAction`, `createRiskMitigation` (commands.go), `organization.go` 의 모든 audit 부착 핸들러
- 현재 동작: gin context 에 `devhub_actor_login` 이 없으면 `X-Devhub-Actor` 요청 헤더 값을 그대로 actor 로 사용. 응답에 `Warning: 299` + `X-Devhub-Actor-Deprecated: true` 헤더 부착하지만 *서버 측 거부는 없음*.
- 영향:
  - 무인증 상태(SEC-2 미해소)에서: 임의 actor 로 organization/command 변경, audit_logs 위조 가능.
  - 인증 상태(SEC-2 해소 후): 인증 컨텍스트가 항상 채워지므로 fallback 도달 불가 — 단, 코드 경로가 남아 있으면 미래 회귀 위험.
- 마감: SEC-2 와 동일 sprint. SEC-2 해소만으로 *현재* 위험은 사라지나, fallback 코드 자체를 prod 에서 제거하거나 dev-only 로 가드해야 한다.

### 완료 정의 (DoD)

1. `requestActor` 의 `X-Devhub-Actor` fallback 을 다음 중 하나로 처리:
   - 옵션 A: 코드 자체를 제거 (가장 안전).
   - 옵션 B: 명시적 환경변수 (`DEVHUB_AUTH_DEV_FALLBACK=1`) 가 켜져 있을 때만 fallback 동작. prod 에서는 무시 또는 401.
2. 해당 헤더가 들어왔을 때 prod 에서 응답에 포함하지 않거나 무시 (현재의 `Warning: 299` 만으로는 부족).
3. 단위 테스트: 인증 컨텍스트가 비었을 때 401 반환 (옵션 A) 또는 dev flag off 시 401 (옵션 B).
4. `// SECURITY (SEC-4)` 주석 제거.

### 의존 / 결합

- SEC-2 와 강결합 — verifier 가 인증 컨텍스트를 항상 채우면 fallback 도달 불가. 단 코드 경로 제거는 회귀 방지.
- audit_logs 의 `actor_login` 컬럼 의미가 신뢰 가능한 식별자로 일관되도록 보장 (감사 무결성).

## 🛡 SEC-5 — DB 에러 메시지의 응답 본문 노출 (defense-in-depth)

- 상태: planned
- 우선순위: medium
- 책임자: backend-core
- 발견 경위: 2026-05-08 코드베이스 전체 보안 리뷰 §5.1. high-confidence 취약점 채택 임계 미달이지만 보안 강화 항목으로 등록.
- 위치: `organization.go`, `commands.go`, `audit.go` 등 다수 핸들러의 `c.JSON(http.StatusInternalServerError, gin.H{"status":"failed","error": err.Error()})` 패턴
- 현재 동작: `pgx` 에러 메시지(스키마/제약/내부 SQL 단편) 가 그대로 5xx 응답 본문에 포함. 자격증명·시크릿·PII 는 포함되지 않으나 외부 공격자에게 스키마 정찰 단서 제공 가능.

### 완료 정의 (DoD)

1. prod 빌드에서 5xx 응답 본문은 일반화된 메시지(예: `internal error`, request id) 로 대체. 상세는 서버 로그로만.
2. 개발 모드는 환경변수 분기로 기존 동작 유지 가능 (debug 친화).
3. 4xx 의 경우(제약 위반·NotFound 등) 에서는 사용자에게 의미 있는 에러를 유지 (의도된 정보).
4. 헬퍼 함수 (`writeServerError`, 또는 기존 `writeStoreError` 확장) 도입으로 패턴 일원화.

### 의존 / 결합

- 코드 변경 범위가 다수 핸들러라 SEC-1~4 와 분리된 별도 PR 권장.
