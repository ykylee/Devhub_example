### T-M0-02 — `X-Devhub-Actor` fallback dev-only env 가드 (SEC-4)

- 우선순위: P0
- 위치: `backend-core/internal/httpapi/commands.go:271-287` (`requestActor`), `backend-core/internal/config/`
- 변경:
  - `DEVHUB_AUTH_DEV_FALLBACK=1` (또는 동등 env) 가 명시되지 않으면 `X-Devhub-Actor` fallback 미동작 → `system` fallback 으로 진행 (또는 401 — T-M0-01 정책 결정에 의존).
  - 동작 시 응답 `Warning: 299` + `X-Devhub-Actor-Deprecated: true` 헤더는 그대로 유지.
  - `// SECURITY (SEC-4)` 주석은 본 task 종료 시 T-M0-11 에서 제거.
- 의존: 없음. (T-M0-01 보다 먼저 가능 — fallback 무효화는 단독 안전 변경)
- DoD:
  1. 환경변수 미설정 + 인증 컨텍스트 비었을 때 `requestActor` 가 `system` 로 fallback (또는 401, 정책 결정 후).
  2. 환경변수 설정 + 인증 컨텍스트 비었을 때 헤더 값을 actor 로 사용 (현재 동작).
  3. 테스트: `auth_test.go` 또는 `commands_test.go` 에 환경별 케이스 추가.
- 위험: dev 환경에서 자동화 스크립트가 헤더 fallback 의존. 변경 후 dev 사용자에게 env 설정 안내 (sprint 종료 시 setup 가이드 갱신).
- 규모: 작음 (~50 LoC, 테스트 포함)

### T-M0-01 — empty `Authorization` 분기 정책 결정 + 화이트리스트 (SEC-2 보강)

- 우선순위: P0
- 위치: `backend-core/internal/httpapi/auth.go:24-44` (`authenticateActor`), `backend-core/internal/httpapi/router.go` 의 라우트 정의
- 변경:
  - 현재 `c.Next()` 분기를 prod 환경에서는 401 로 바꿈.
  - dev 환경(예: `DEVHUB_AUTH_DEV_FALLBACK=1`) 에서는 *명시적 화이트리스트* 라우트만 통과시키고 그 외는 401.
  - 화이트리스트 후보: `/health`, `/api/v1/snapshot/*` (대시보드 메트릭 — 무인증 노출 가능 여부 확인 필요), `/api/v1/integrations/gitea/webhooks` (자체 HMAC 시그니처 검증 사용).
  - 정책 결정: 화이트리스트는 **명시적 nopath array 또는 별도 router group** 으로 분리해 미들웨어 스코프 자체를 좁힌다.
- 의존: T-M0-02 (env 변수 표준 일관)
- DoD:
  1. `/api/v1/users`, `/api/v1/admin/*`, `/api/v1/audit-logs`, `/api/v1/organization/*`, `/api/v1/realtime/ws` 가 무 Authorization 으로 401 반환.
  2. `/health`, `/api/v1/integrations/gitea/webhooks` 는 그대로 200 (HMAC 시그니처가 있는 webhook 제외).
  3. snapshot 류 라우트 (`/api/v1/dashboard/metrics`, `/api/v1/infra/*`, `/api/v1/repositories`, `/api/v1/risks*`, `/api/v1/ci-runs*`) 의 인증 정책 결정 — *프론트가 토큰을 항상 부착하도록* 통일하면 401 유지가 자연.
  4. 통합 테스트: 화이트리스트 / 보호 라우트 매트릭스.
- 위험:
  - snapshot 라우트가 현재 frontend 에서 무토큰 호출 가능성. T-M0-08 (frontend 토큰 부착) 와 동기화 필요.
  - Webhook 라우트는 자체 HMAC 검증을 하므로 미들웨어 우회로 두는 것이 안전 — 그러나 미들웨어 적용 그룹과 별도 그룹으로 분리하는 router 변경이 필요.
- 규모: 중 (~150 LoC + 테스트, router.go 구조 일부 재배치)

### T-M0-03 — Role 가드 미들웨어 구현 (SEC-3)

- 우선순위: P0
- 위치: `backend-core/internal/httpapi/authz.go` (신규) 또는 `auth.go` 확장
- 변경:
  - `requireRole(roles ...domain.AppRole) gin.HandlerFunc` 헬퍼.
  - 컨텍스트의 `devhub_actor_role` 을 읽어 `defaultRBACPolicy()` 의 role rank 와 비교 — 부족 시 403 + audit.
  - `manager+` 같은 *최소 rank* 비교 헬퍼 (`requireMinRole(domain.AppRoleManager)`).
- 의존: T-M0-01 (authenticated context 정합성), 향후 T-M0-06 (verifier 가 role 을 정확히 채움)
- DoD:

  2. 403 응답 시 audit_logs 에 `auth.role_denied` 로 기록 (best-effort).
  3. `// SECURITY (SEC-3)` 주석은 T-M0-04 종료 시점에 제거.
- 위험: rank 비교 로직 명세 — `defaultRBACPolicy()` 의 permission rank (none=0, read=10, write=20, admin=30) 와 role-resource matrix 의 행 단위 매핑 정합 필요.
- 규모: 중 (~100 LoC + 테스트)

### T-M0-04 — Role 가드 라우트 적용 (SEC-3)

- 우선순위: P0
- 위치: `backend-core/internal/httpapi/router.go` 의 `/api/v1` 라우트 정의
- 변경 — 5 라우트 클래스 매핑:
  - `/admin/*` (`POST /admin/service-actions` 등) → `system_admin`
  - 사용자 mutation: `POST/PATCH/DELETE /api/v1/users[/:id]` → `system_admin`
  - 감사 이력: `GET /api/v1/audit-logs` → `manager` 이상
  - 조직 단위 CRUD: `POST/PATCH/DELETE /api/v1/organization/units...`, `PUT /api/v1/organization/units/:unit_id/members` → `system_admin`
  - command 생성: `POST /api/v1/admin/service-actions`, `POST /api/v1/risks/:risk_id/mitigations` → `manager` 이상
- 의존: T-M0-03 완료
- DoD:
  1. router.go 의 변경된 라인이 `requireMinRole`/`requireRole` 미들웨어를 라우트별로 부착.
  2. `// SECURITY (SEC-3)` 주석 제거.
  3. 통합 테스트: 각 role 별 5 라우트 클래스 PASS/403 매트릭스.
- 위험: 라우트별 미들웨어 부착이 router 의 재구조화를 요구할 수 있음 (group 단위 vs 단일 핸들러 wrap). gin 패턴상 둘 다 가능하므로 단일 핸들러 wrap 으로 시작.
- 규모: 중 (~50 LoC router 변경 + 테스트 매트릭스 추가)
