## 1. 현재 코드베이스 진단 (2026-05-11)

### 1.1 envelope (T-M1-02)

- 직전 sprint 의 신규 핸들러 (`auth_login`/`auth_token`/`auth_logout`/`auth_signup`/`accounts_admin`/`auth_consent`) 는 `{status, data, error?}` envelope 을 사용함 ✓
- 다만 기존 핸들러 (`commands.go`/`organization.go`/`rbac.go`/`audit.go`/`infra.go`/`risks.go` 등) 의 envelope 일관성 미점검. 특히 list 응답이 `data` 안에 array 인지 top-level array 인지 라우트별로 상이할 가능성.
- `docs/backend_api_contract.md` 의 envelope 정의를 source-of-truth 로 일괄 검증.
- role wire format 은 backend `developer | manager | system_admin` 로 정상 (M1-PR-G2 머지 시점부터). 잔존 `DEVELOPER|MANAGER|ADMIN` 확인은 grep 으로 검증.

### 1.2 command lifecycle (T-M1-03)

- `domain/command.go` 의 상태 enum 확인 필요. 6 상태 (`pending | running | succeeded | failed | rejected | cancelled`) 모두 정의되었는지, 핸들러/worker 가 일관 사용하는지.
- dry-run 경계: `commandworker.Worker` (dry-run) + `commandworker.LiveWorker` (live) 가 main.go 에 분리 주입되어 있음 ✓ — 단, dry-run 의 store 영속화 정책이 spec 과 일치하는지 (dry-run 도 store 에 기록하는지 vs 응답만으로 종료) 확인.

### 1.3 auth_test 매트릭스 (T-M1-05)

- `auth_test.go` 와 `authz_test.go` 가 이미 존재. 기존 케이스 수와 spec 의 16+ 매트릭스 차이 점검.
- DEVHUB_ENV=prod + verifier nil 시 startup 거부 통합 테스트 필요.

### 1.4 frontend types split (T-M1-07)

- `frontend/lib/services/types.ts` 가 단일 파일에 wire + UI 형태 혼재 추정. 컴포넌트가 직접 wire 타입 import 하는지 확인.
- `identity.service.ts` 는 이미 `ROLE_BACKEND_TO_UI` / `ROLE_UI_TO_BACKEND` 변환 보유 — 다른 service 들도 동일 패턴인지 확인.

### 1.5 WebSocket envelope (T-M1-08)

- `internal/httpapi/realtime.go` + `RealtimeHub` 의 publish 메시지 형태 확인. `command.status.updated` 만 publish 중이며 envelope 5 키 (`schema_version, type, event_id, occurred_at, data`) 를 모두 채우는지 점검.
- frontend `realtime.service.ts` / `websocket.service.ts` 의 decode 로직도 5 키 기준으로 통합.
