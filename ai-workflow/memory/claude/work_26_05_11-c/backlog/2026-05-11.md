## 1. 현재 코드베이스 진단

### 1.1 audit_logs schema (현재)

`backend-core/migrations/000003_create_command_audit_tables.up.sql:23-33`:
```sql
CREATE TABLE audit_logs (
    id BIGSERIAL PRIMARY KEY,
    audit_id TEXT NOT NULL UNIQUE,
    actor_login TEXT NOT NULL,
    action TEXT NOT NULL,
    target_type TEXT NOT NULL,
    target_id TEXT NOT NULL,
    command_id TEXT REFERENCES commands(command_id) ON DELETE SET NULL,
    payload JSONB NOT NULL DEFAULT '{}'::jsonb,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

→ 추가할 컬럼: `source_ip TEXT`, `request_id TEXT`, `source_type TEXT`. 마지막 마이그레이션 번호 `000007` 이므로 신규는 `000008_audit_logs_actor_enrichment`.

### 1.2 audit handler / store

- `internal/httpapi/audit.go` 의 `recordAudit` / `recordAuditBestEffort` 가 audit row 빌드 — actor_login, action, target_*, payload 만. 신규 3 필드 채움 분기 추가 필요.
- `internal/store/postgres.go` 의 `CreateAuditLog` INSERT — 컬럼 3개 추가 + scan.
- domain `AuditLog` struct — 필드 3개 추가.

### 1.3 request_id 인프라 부재

- 현재 `gin.Context` 에 request_id 없음. 응답 헤더 `X-Request-ID` 도 없음.
- 신규 미들웨어 — `requireRequestID(c *gin.Context)` 가 들어오는 `X-Request-ID` 또는 신규 생성하여 `c.Set("devhub_request_id", id)` + `c.Header("X-Request-ID", id)`.
- `authenticateActor` 보다 먼저 (또는 동시에) v1 group 에 install.

### 1.4 source_type 분류

- `oidc`: Bearer token verifier 가 actor 도출 (정상 인증 흐름)
- `webhook`: Gitea webhook (`/api/v1/integrations/gitea/webhooks`) — bypass route
- `system`: dev fallback 또는 backend internal call

### 1.5 writeServerError 통합

`internal/httpapi/errors.go:writeServerError` 가 현재 `op` + `err` 만 로그. request_id 추가 필요.
