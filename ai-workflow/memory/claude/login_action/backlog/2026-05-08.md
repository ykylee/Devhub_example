  - `frontend/lib/auth/pkce.ts` (신규) — `code_verifier` + `code_challenge(S256)` 생성/검증
  - `frontend/lib/services/identity.service.ts` 수정 — `jsonRequest` 가 Authorization 헤더 자동 부착
  - `frontend/lib/services/rbac.service.ts` + 기타 service 수정 — 동일 헬퍼 사용으로 통합
  - `frontend/app/login/page.tsx` 수정 — `code_challenge` + `code_verifier` 생성/저장 후 Hydra 로 보냄
  - `frontend/components/layout/AuthGuard.tsx` 수정 — `whoAmI` 가 token 없을 때 즉시 `/login` redirect

- DoD (안)
  1. PKCE: frontend 가 `code_challenge_method=S256` + `code_verifier` (sessionStorage) 생성
  2. Hydra `/oauth2/token` (public client, no client_secret) 직접 호출 — `grant_type=authorization_code`, `code`, `code_verifier`, `redirect_uri`, `client_id`
  3. token 응답 파싱: `access_token` (30m), `refresh_token` (720h, offline_access scope), `id_token`
  4. Token 저장 — sessionStorage + 메모리 캐시 (XSS 위험 인지 후 결정)
  5. `state` 검증 — sessionStorage 의 `oauth_state` 와 callback 쿼리의 `state` 일치
  6. Authenticated fetch — `Authorization: Bearer <access_token>` 자동 부착, 401 시 refresh 시도 또는 `/login` redirect
  7. Refresh token rotation (선택, 후속 cleanup)
  8. `npm run build` PASS + 신규 unit test (token-store, pkce)
  9. e2e: `/login` → password → `/developer` 진입까지 PASS

- 위험 / 결정 대기
  - **Token 보안 정책**: sessionStorage (XSS 위험 노출) vs httpOnly cookie via BFF (backend 가 token 을 쥐고 frontend 는 cookie 만 통과). ADR-0001 의 7.2 는 frontend 직접 보유 가정. SPA 표준 + PKCE 가정에서 sessionStorage 가 ADR 일치. **PR-LOGIN-3 진입 시 사용자 결정 권장**.
  - **CSP**: token 저장 방식 결정 후 CSP 헤더 (frame-ancestors, script-src) 강화 필요.

## 4. 결정 / 위험

- **Token 저장 위치 (PR-LOGIN-3 진입 시 결정)**: sessionStorage vs BFF cookie. SPA 표준은 sessionStorage + 짧은 access_token 수명 + refresh rotation. 사용자 환경 보안 요구가 높으면 BFF.
- **Refresh token rotation**: 1차 미도입, 후속 hygiene 가능.
- **MFA**: ADR-0001 §8.3 결정 — 1차 미도입. PR-LOGIN-1~4 범위 밖.
- **외부 client 등록 (consent UI 필요)**: ADR-0001 §8.2 — 1차 first-party only, silent consent. 외부 SaaS 진입 시 별도 phase.
