# Auth/RBAC 의사결정에서 얻은 교훈

- 상태: draft
- 최종 수정일: 2026-05-10
- 원천 문서: `docs/adr/0001-idp-selection.md`, `docs/adr/0002-rbac-policy-edit-api.md`, `docs/architecture.md`

## 왜 이 주제가 중요했나

인증(Auth)과 권한(RBAC)은 보통 기능이 커진 뒤에 손보게 됩니다.  
하지만 이 순서는 거의 항상 비용이 큽니다. 구현이 먼저 퍼지고 나면, 이후 정비 단계에서

- API 계약과 실제 권한 동작 불일치
- UI 노출 정책과 서버 권한 정책 불일치
- 운영 감사 로그의 책임 경계 불명확

같은 문제가 반복됩니다.

## 교훈 1: 인증 경계는 "나중"이 아니라 "초기"에 고정해야 한다

핵심은 "누가 요청했는가"를 신뢰 가능한 방식으로 먼저 확정하는 것입니다.

- 헤더 기반 임시 경계는 개발 편의에는 유용하지만 운영 경계로 남기면 위험
- 세션/JWT/OIDC 등 신뢰 체계를 먼저 정하고, 이후 기능을 그 위에 얹어야 함

즉, 인증은 기능의 부속품이 아니라 모든 기능의 전제입니다.

## 교훈 2: 문서 계약과 런타임 enforcement는 반드시 같은 모델이어야 한다

권한 정책 편집 UI가 존재해도 서버 enforcement가 정적 코드에 묶여 있으면,
사용자는 "저장했는데 왜 적용 안 되지?"를 경험하게 됩니다.

이 상황은 UX 문제를 넘어 운영 신뢰성 문제입니다.  
그래서 정책 모델, API, 서버 권한 체크가 같은 기준으로 움직이도록 맞춰야 합니다.

## 교훈 3: 권한은 "노출"과 "실행"을 분리해 설계해야 한다

- UI 노출: 사용자에게 어떤 메뉴/액션을 보여줄지
- 서버 실행: 실제 요청을 허용/거부할지

이 둘은 모두 필요합니다.  
UI에서 숨겼다고 서버 검증이 면제되면 안 되고, 서버만 막아도 UX 혼란은 남습니다.

## 교훈 4: ADR은 결과 문서가 아니라 의사결정 자산이다

좋은 ADR은 "무엇을 선택했는가"보다

- 왜 그 선택을 했는지
- 어떤 대안을 배제했는지
- 어떤 비용과 리스크를 감수했는지

를 남깁니다. 그래야 후속 팀이 같은 논쟁을 반복하지 않습니다.

## 실무 적용 체크리스트

1. 인증 주체 식별 경로가 문서/코드에서 동일한가
2. RBAC 정책 모델과 서버 enforcement 로직이 동일한가
3. UI 메뉴 노출 정책과 API 권한 정책이 동기화되어 있는가
4. 감사 로그가 권한 경계(누가, 무엇을, 왜)를 추적할 수 있는가

## 마무리

Auth/RBAC는 기능 확장의 마지막 단계가 아니라, 제품 신뢰를 만드는 첫 단계입니다.  
DevHub의 교훈은 단순합니다. **권한 모델은 설계 문서에서 끝내지 말고, 런타임 동작으로 증명해야 한다**는 것입니다.
