# 역할별 UX를 "진입 우선순위"로 설계한 이유

- 상태: draft
- 최종 수정일: 2026-05-10
- 원천 문서: `docs/requirements.md`, `docs/development_roadmap.md`, `docs/architecture.md`

## 문제: 역할별 전용 화면은 빨리 복잡해진다

초기 제품에서는 역할마다 화면을 완전히 분리하는 접근이 직관적으로 보입니다.  
하지만 실제 운영 단계에 들어가면 다음 문제가 반복됩니다.

- 공통 기능(알림, 검색, 계정, 조직 컨텍스트)이 역할별로 중복 구현됨
- 작은 정책 변경도 여러 화면을 동시에 수정해야 함
- 역할이 늘어날수록 QA 범위가 선형이 아니라 조합적으로 증가함

결국 "역할별 화면 분리"는 빠른 시작에는 유리하지만, 장기 운영 비용이 높아지는 패턴으로 이어집니다.

## 선택: 역할별 UX는 "기본 진입 우선순위"로 제공

DevHub는 역할별 UX를 완전 분리 대신 **역할별 기본 진입 페이지 우선순위**로 제공합니다.

- `developer` → 개발 대시보드
- `manager` → 관리 대시보드
- `system_admin` → 시스템 대시보드 + 시스템 설정

핵심은 "역할별 첫 화면"만 다르게 시작하고, 공통 영역은 가능한 한 하나의 정보 구조로 유지하는 것입니다.

## 왜 이 접근이 실용적인가

1. 제품 일관성
- 공통 컴포넌트와 상호작용 패턴을 공유할 수 있습니다.
- 사용자 이동 경로가 단순해지고 온보딩 비용이 줄어듭니다.

2. 운영 효율
- 기능 변경 시 수정 지점이 줄어듭니다.
- 문서 계약(요구사항/아키텍처/로드맵)과 실제 UI를 맞추기 쉬워집니다.

3. 확장성
- 새로운 역할이 추가되어도 "우선 진입 경로"만 정의하면 시작할 수 있습니다.
- 전체 정보 구조를 다시 뜯어고치지 않아도 됩니다.

## 함께 지켜야 할 원칙

진입 우선순위 전략이 성공하려면 두 가지가 함께 필요합니다.

- 권한 기반 노출 제어: 특히 시스템 운영 메뉴는 `system_admin`으로 명확히 분리
- 계약-구현 정합성: 문서상 역할 정의와 실제 라우팅/메뉴 노출이 항상 일치

## 마무리

DevHub의 역할 기반 UX는 "역할마다 다른 제품"을 만드는 전략이 아닙니다.  
같은 제품 안에서 각 역할의 시작점을 다르게 설계해, 일관성과 확장성을 동시에 확보하는 전략입니다.
