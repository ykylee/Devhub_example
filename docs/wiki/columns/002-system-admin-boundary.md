# 시스템 관리자 권한 경계를 분리해야 하는 이유

- 상태: draft
- 최종 수정일: 2026-05-10
- 원천 문서: `docs/requirements.md`, `docs/architecture.md`, `docs/backend_api_contract.md`

## 시스템 기능은 "관리 기능"과 성격이 다르다

일반 관리 기능은 주로 업무 가시성과 협업 효율을 다룹니다.  
반면 시스템 운영 기능은 서비스 가용성, 보안, 운영 안정성에 직접 영향을 줍니다.

예를 들어 다음 작업은 실패 비용이 큽니다.

- 운영 설정 변경
- 인프라 제어 액션
- 보안 관련 정책 조정

이런 기능은 일반 관리자 권한과 같은 계층으로 두면, 권한 경계가 쉽게 흐려집니다.

## DevHub의 경계 설계

DevHub는 시스템 운영 영역을 별도 경계로 분리합니다.

1. 시스템 대시보드/시스템 설정은 `system_admin`에게만 노출
2. 일반 관리 대시보드와 시스템 운영 메뉴를 구조적으로 분리
3. 운영 액션은 권한 검사와 감사 로그를 함께 설계

즉, 메뉴 노출 단계부터 실행 권한 단계까지 동일한 정책을 적용합니다.

## 기대 효과

- 오남용 위험 감소: 실수로 인한 운영 영향 최소화
- 책임 명확화: 누가 어떤 수준의 변경을 할 수 있는지 분명해짐
- 감사 가능성 강화: 운영/보안 이벤트를 일관되게 추적 가능

## 구현에서 중요한 체크포인트

- UI: 권한 없는 사용자에게 시스템 메뉴를 애초에 노출하지 않기
- API: 노출 제어와 별개로 서버에서 항상 권한 검증 수행
- 문서: 요구사항/아키텍처/계약 문서의 권한 정의를 동일하게 유지

## 마무리

권한 분리는 UX 제약이 아니라 운영 안정성을 위한 제품 설계입니다.  
DevHub는 이 경계를 "보이지 않는 규칙"이 아니라, 메뉴 구조와 실행 정책에 드러나는 형태로 유지합니다.
